# 封禁与举报功能实施计划

## 项目背景
当前系统只有删除功能，缺乏更细粒度的内容管理功能。需要增加封禁功能（可解禁）和完善举报功能。

## 当前状态分析

### 1. 数据库现状
- **post表**：已有status字段（0正常，1审核中，2封禁），is_deleted字段（0正常，1软删除）
- **novel表**：已有audit_status字段（0待审核，1审核通过，2审核拒绝），is_deleted字段（0正常，1软删除）
- **report表**：已存在但功能不完整

### 2. 代码现状
- **PostController**：有删除功能，无封禁功能
- **NovelController**：有删除功能，无封禁功能  
- **ReportController**：只有模拟功能，未实际实现
- **前端页面**：有举报按钮但功能未实现

## 功能设计方案

### 1. 封禁功能设计

#### 状态定义
```
帖子状态（post.status）：
- 0: 正常
- 1: 审核中
- 2: 封禁（新增）
- 3: 草稿（可选）

小说审核状态（novel.audit_status）：
- 0: 待审核
- 1: 审核通过
- 2: 审核拒绝
- 3: 封禁（新增）
```

#### 功能特性
1. **可逆操作**：封禁后可解禁，删除后不可恢复
2. **权限控制**：仅管理员可执行封禁/解禁操作
3. **内容过滤**：封禁内容对普通用户不可见
4. **消息通知**：封禁/解禁时通知作者

#### API设计
```
POST /post/ban/{id}        # 封禁帖子
POST /post/unban/{id}      # 解禁帖子
POST /novel/ban/{id}       # 封禁小说
POST /novel/unban/{id}     # 解禁小说
```

### 2. 举报功能集成

#### 举报流程
1. 用户在帖子/小说详情页点击举报按钮
2. 弹出举报表单（选择举报类型、填写原因）
3. 提交举报到后台
4. 管理员在后台处理举报

#### 举报类型
- 违规内容
- 垃圾广告
- 人身攻击
- 色情低俗
- 其他

#### API设计
```
POST /report/create        # 创建举报
GET  /report/list          # 获取举报列表（管理员）
POST /report/process/{id}  # 处理举报
```

## 技术实现细节

### 1. 封禁功能实现
```java
// PostService新增方法
public boolean banPost(Long postId) {
    Optional<Post> postOpt = postRepository.findById(postId);
    if (postOpt.isPresent()) {
        Post post = postOpt.get();
        post.setStatus(2); // 2表示封禁
        post.setUpdateTime(LocalDateTime.now());
        postRepository.save(post);
        
        // 发送通知
        sendNotification(post.getAuthorId(), "您的帖子已被封禁");
        return true;
    }
    return false;
}

public boolean unbanPost(Long postId) {
    Optional<Post> postOpt = postRepository.findById(postId);
    if (postOpt.isPresent()) {
        Post post = postOpt.get();
        post.setStatus(0); // 0表示正常
        post.setUpdateTime(LocalDateTime.now());
        postRepository.save(post);
        
        // 发送通知
        sendNotification(post.getAuthorId(), "您的帖子已解禁");
        return true;
    }
    return false;
}
```

## 风险评估与应对

### 1. 技术风险
- **风险**：现有查询逻辑需要修改以过滤封禁内容
- **应对**：在Service层统一处理过滤逻辑，确保一致性

### 2. 用户体验风险
- **风险**：举报流程复杂可能降低用户使用意愿
- **应对**：简化举报表单，提供常用举报类型选项

### 3. 性能风险
- **风险**：举报数量增多可能影响查询性能
- **应对**：实现分页查询，添加索引优化

## 成功标准

1. 管理员可以封禁/解禁帖子和小说
2. 用户可以举报违规内容
3. 管理员可以处理举报并执行相应操作
4. 封禁内容对普通用户不可见
5. 系统性能无明显下降
